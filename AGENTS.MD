# AGENTS.md

Context for AI coding agents (Claude Code, etc.) working with this repository.

## About the Project

Data Room MVP — a take-home assignment. A SPA for organizing folders and files (similar to Google Drive/Dropbox with a single root directory). Priorities: UX and handling edge cases > design > code cleanliness.

## Tech Stack

- Vite + React + TypeScript
- Tailwind CSS v4
- shadcn/ui (Radix, Sera preset)
- Zustand — application state (metadata) with `persist` middleware
- IndexedDB via `idb` — Blob file storage as a mock backend
- IndexedDB via `idb-keyval` — persisted store metadata (survives page reloads)
- Sonner — toast notifications after every action (create/rename/delete/upload/download)
- No React Compiler, no Next.js — pure client-side SPA, SSR is not required

## UI

- Use Lucide React for all icons.
- Use shadcn/ui components for all UI elements (Card, Button, etc.) with the Sera preset.

## Code Style

- Use `interface` for data models (`FolderNode`, `FileNode`) instead of `type` — the project already follows this convention, keep it consistent.
- Event handlers should use the form `const handleX = () => {}` instead of function declarations inside components.
- Always explicitly specify return types for hooks and functions.
- Never use `any`; use `unknown` only when the type is genuinely unknown.
- Validation and business rules belong exclusively in `lib/*Helpers.ts`. Components should call these helpers rather than implementing the logic themselves.
- Imports order:
  1. External packages
  2. `@/` aliases
  3. Relative imports

  Separate each group with an empty line.

## Data Model

Use a flat normalized structure — **not** nested objects with `children: []`.

```ts
interface BaseNode {
  id: string
  name: string
  normalizedName: string // lowercase + trimmed for case-insensitive duplicate detection
  parentId: string | null // null = root
  type: "folder" | "file"
  createdAt: number
  updatedAt: number // used for "Updated" sorting
}

interface FolderNode extends BaseNode {
  type: "folder"
}

interface FileNode extends BaseNode {
  type: "file"
  mimeType: string
  size: number
  extension: string // e.g. "pdf" to avoid split(".") in components
  blobId: string // IndexedDB key; Blob itself is NOT stored in Zustand
}

type DataRoomNode = FolderNode | FileNode

type SortOption =
  | "name-asc"
  | "name-desc"
  | "updated-asc"
  | "updated-desc"
  | "size-asc"
  | "size-desc"

type ViewMode = "list" | "grid"

type FoldersPosition = "top" | "bottom"

type TypeFilter = "all" | "folder" | "file"
```

The store consists of:

- `Record<string, DataRoomNode>`
- `currentFolderId`
- UI settings (`sortOption`, `viewMode`, `foldersPosition`, `typeFilter`, `isHydrated`, `isUploading`)

Child nodes are computed on demand:

```ts
nodes.filter((n) => n.parentId === currentFolderId)
```

Do **not** duplicate them in `childrenIds`.

Persist UI settings using `partialize`, excluding `isHydrated` and `isUploading` (they are ephemeral).

## Responsibility Separation

- **`store/dataRoomStore.ts`** — stores node metadata and UI state (sorting, filtering, view mode, etc.). Never stores Blob data. Persist using `idb-keyval`, not `localStorage`.
- **`store/conflictStore.ts`** — manages the queue of filename conflict dialogs during uploads. Not persisted. Each conflict is represented as a `Promise` resolved by the user's decision (`overwrite` / `copy` / `cancel`).
- **`db/blobStore.ts`** — the only place that interacts directly with IndexedDB (`idb`) for Blob storage by `blobId`. Components and stores must never access `idb` directly.
- Keep two separate IndexedDB modules:
  - `idb` (via `blobStore.ts`) for binary Blob files.
  - `idb-keyval` (inside the `dataRoomStore.ts` persist adapter) for serialized metadata.
  Do not mix them.
- **`contexts/`** — React contexts used to avoid prop drilling. Example: `CurrentFolderContext` wraps the `useCurrentFolder` hook.
- **`node/*`** — shared components for files and folders (context menu, rename, delete). They work with `DataRoomNode`, branching on `type` only where necessary. Do not duplicate rename/delete logic separately for files and folders.
- Business logic (cascade deletion, duplicate resolution, downloads, etc.) belongs in `lib/`, not in components. Functions should remain pure and independently testable.

## Required Edge Cases

- Duplicate file/folder name in the same directory → automatically append a suffix like `name (1).pdf` (see `lib/nameHelpers.ts`).
- Uploading a non-PDF file → show a clear error instead of silently ignoring it.
- Deleting a folder → recursively delete all descendants and show a confirmation dialog with the exact number of nested folders and files:
  > "This folder contains 12 folders, 43 files."
- Creating or renaming with an empty name → trim input, validate it, disable submission when invalid, and reject whitespace-only strings.
  - `"   "` → error
  - `" Report "` → `"Report"`
- Names longer than 255 characters → validation error.
- Renaming to an existing name in the same folder → apply the same automatic suffix logic used for duplicate uploads.
- Empty folder → display the `EmptyState` component instead of a blank screen.
- Long filenames → use `text-overflow: ellipsis` with a tooltip.
- Multi-file upload (multi-select/drag-and-drop) → process every file independently. One invalid file (non-PDF/corrupted) must not block the others. Show an individual toast notification for each file.

## UI Features Beyond Basic CRUD

These are inexpensive to implement and significantly improve the perceived quality of the application, so they are included in the main scope:

- **Toast notifications (Sonner)** — after every action: create/delete/rename folder, successful/failed upload, download. During batch uploads, show one toast per file.
- **Sort** — dropdown with:
  - Name ↑/↓
  - Updated ↑/↓
  - Size ↑/↓

  Implement logic in `lib/sortHelpers.ts` as a pure function using:

  - `name-asc`
  - `name-desc`
  - `updated-asc`
  - `updated-desc`
  - `size-asc`
  - `size-desc`

- **TypeFilter** — `all` / `folder` / `file`
- **ViewMode** — `list` / `grid`
- **FoldersPosition** — `top` / `bottom` (folders first or mixed with files)
- **URL sync (`useUrlSync`)** — synchronize navigation state with query parameters:

  ```
  ?folder=id&sort=name-asc&view=grid&filter=all&folders=top
  ```

  Support browser Back/Forward navigation via `popstate`. Apply only once after store hydration.

- **Clickable breadcrumbs** — every path segment is a link that changes `currentFolderId`.
- **Context menu (right click)** — Rename/Delete using the same component as the "..." menu. Do not duplicate logic.
- **Loading state** — minimal loading indicators for asynchronous operations (uploads and initial IndexedDB read).

## Important Implementation Details

- **`validateName`** returns a discriminated union:

```ts
{ ok: false; error: string } | { ok: true; value: string }
```

Do not replace it with:

```ts
{ ok: boolean; error?: string; value?: string }
```

That breaks TypeScript narrowing.

- **`conflictStore`**
  - `promptConflict()` returns `Promise<ConflictDecision>`.
  - The promise resolves only when the user closes the dialog.
  - The dialog component calls `resolveConflict(id, decision)`.

- **`useUrlSync`**
  - Use it only once at the App/Layout level.
  - It tracks the previous `folderId` with `useRef` to distinguish:
    - `pushState` → navigation
    - `replaceState` → settings changes

- **`lib/utils.ts`**
  - Contains only `cn()` (the standard shadcn utility for merging class names).
  - Do not add business logic there.

- **`lib/downloadHelper.ts`**
  - `downloadNode()` downloads files using `URL.createObjectURL`.
  - All Blob handling must go through `blobStore.ts`.

## What NOT to Do

- Do not store Blob/File objects in Zustand or `localStorage` (storage limits and main-thread blocking).
- Do not implement authentication, deployment, search, or other extra features until the required functionality (folders, files, edge cases) is complete.
- Do not manually edit `components/ui/*`; regenerate them using the shadcn CLI if necessary.
- Do not introduce an FSD architecture (entities/features/widgets); it's unnecessary overhead for a project of this size.
- Do not access `idb` outside `blobStore.ts` or `idb-keyval` outside `dataRoomStore.ts`.
